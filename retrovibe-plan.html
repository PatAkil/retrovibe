▚ Retrovibe — MVP Plan

Describe a retro game in Claude → skills turn it into a playable, high-quality game. Laptop loop: create → play → iterate → reset → repeat. Deploy comes post-MVP. Rev 4 — hardened across three adversarial review rounds (see §08).

Settled decisions

Stack: TypeScript + Canvas 2D, no game framework. A small transparent engine in this repo encodes the quality rules as primitives; skills carry all game-making knowledge and all commands — no helper scripts. Every game is a self-contained folder cloned from a template:

cp -r workspace/game-template workspace/<game-name>   # only after the collision guard in §04 passes

Engine is copied into each game, not shared. Every game folder is standalone and independently deployable later (npm run build → dist/ with base:'./' so it works from any subpath, e.g. /games/my-game/); later engine changes can never break older games; creation stays a pure cp -r. Canonical source lives in game-template/engine/.
One install at the root. Per-game npm run dev / npm run build work because npm prepends every ancestor node_modules/.bin to PATH — no install step per game. Vite's cacheDir is pointed at the repo root so running a game never creates a node_modules/ inside it.
Git: git init + one initial commit as the clean-slate baseline; the orchestrator commits each finished game as a checkpoint (so reset is recoverable). Template integrity is enforced, not assumed: before every clone and every reset, git status --porcelain workspace/game-template must be clean — if dirty, restore with git checkout -- workspace/game-template && git clean -fd workspace/game-template before proceeding (checkout reverts tracked changes/deletions; clean removes untracked files, which checkout cannot — and it respects .gitignore, so build cruft is untouched). A root .gitignore (node_modules/, dist/, .vite/) keeps the baseline commit small and these checks fast; checkpoint commits are scoped to workspace/<game-name>, never -A. No remote.
Build is necessary, not sufficient. A green tsc/vite build proves compilation only. Every handoff to the user also requires the runtime smoke check in §05, and the user — not Claude — is the real playtester. Claude never claims to have "played" the game.

02Engine modules — quality rules as defaults, not conventions

ModuleContents
loop.tsFixed-timestep loop (accumulator), update(dt)/render split, pause on blur. Frame-time clamp is mandatory: cap the per-frame delta (~250 ms) and reset the clock on focus, independent of the pause flag — otherwise alt-tab/sleep produces a spiral-of-death catch-up or physics teleport on return
input.tsUnified keyboard map (arrows+WASD → direction; Space/Z/X/Enter → actions), pressed/held/released edges, stuck-key handling on blur. First keydown also unlocks audio (see audio.ts)
scenes.tsEnforced state machine TITLE → PLAYING ⇄ PAUSED → (GAME_OVER | WIN) → restart. Level advance = PLAYING re-entry. Invariant (checked by improving-game-quality, not encoded as the only terminal): the game is always playable and a lose state is always reachable
draw.tsPixel-scaled rendering, ASCII-art sprite maps → colored pixel sprites, retro text
palette.tsCurated retro palettes (PICO-8-style) + palette-swap support (feeds easter eggs)
particles.tsAmbient background presets (stars, rain, snow, embers, bubbles) + burst emitter for impacts
juice.tsScreen shake, screen flash, hit-stop / freeze frames
audio.tsWebAudio chiptune synth presets: jump, pickup, explosion, hit, UI blip — zero asset files. Autoplay policy handled by design: the AudioContext is created/resumed lazily inside the first user-gesture handler (first keydown on the title screen), never at module load — the TITLE→PLAYING keypress is the documented unlock point
ui.tsHUD helpers (score/lives/text) with enforced SAFE_MARGIN from viewport edges
crt.tsRetro screen filter: scanlines + vignette + subtle flicker
runtime.tsHost contract with a pinned wire format: messages are { source: 'retrovibe', type: 'gameOver' | 'scoreChanged' | 'stateChanged', payload }. Embed detection is window.parent !== window; when embedded, postMessage to parent with an explicit targetOrigin (default '*' for MVP, tighten at deploy); otherwise log to console. Verified against a tiny parent-frame harness (§06) so the postMessage path has real coverage in MVP

Your four rules, mapped: UI margins → ui.ts always-playable loop + lose condition → scenes.ts background particles → particles.ts screen shake → juice.ts

03Template game

A minimal complete game (target ~150 lines — a soft target, correctness wins over line count) inside game-template/game/ proving every rule — and serving as the reference implementation all skills point to: title screen → play (move a ship, collect pickups, dodge a hazard; contact = lose) → game over → restart. Starfield, shake+flash+burst on hit, chiptune sfx (unlocked on first keypress), safe-margin HUD, CRT filter, runtime messaging. The template is never run in place — playtest clones, not the template.

04The skills — eleven; the orchestrator's routing list is the single source of truth (CLAUDE.md links to it, never duplicates it)

creating-a-game — the orchestrator:
  1. Elicit (gate): if the request is vague, invoke helping-the-user first. Bar to proceed: core loop, lose condition, and controls are known.
  2. Route: derive the folder-friendly name and check whether workspace/<game-name> exists. Exists → hand off to iterating-on-a-game (or ask: new name / iterate / explicit overwrite). Never run cp -r onto an existing folder — it nests the template inside it and corrupts the game.
  3. Setup workspace: verify template integrity (git status --porcelain workspace/game-template clean, restore if not), then cp -r workspace/game-template workspace/<game-name>. No dependency install needed.
  4. Develop: read the copied game files first; modify to fulfill the request. Use the companion skills: messaging-game-over, ensuring-arcade-visuals, handling-user-input, improving-game-quality, adding-easter-egg — and building-platformer-games if it's a platformer.
  5. Validation loop: after every edit, fast typecheck: cd workspace/<game-name> && npm run check (tsc --noEmit). Before notifying the user: full npm run build plus the runtime smoke check from playing-the-game. A green build alone never triggers "done".
  6. Checkpoint: commit the finished game (git add workspace/<game-name> && git commit) so reset is recoverable.
iterating-on-a-game — edits an existing game in place ("make the ship faster"): never clones, runs the same validation loop + smoke check + checkpoint commit as the orchestrator. The default path for any request naming a game that already exists. Inverse guard (closes the routing loop with the orchestrator): if workspace/<name> does not exist — after a reset, or a name mismatch — never edit a missing folder; hand off to creating-a-game (or ask the user which existing game they meant).
helping-the-user — behavior-elicitation questions, combining described behaviors into a coherent core loop, scoping guidance. Invoked as the orchestrator's Step 1 gate.
ensuring-arcade-visuals — look only: palette discipline, pixel scale, CRT filter integration. (Readability/margins live in improving-game-quality — scopes are disjoint by design.)
handling-user-input — the unified keyboard contract, edge-vs-held semantics, control hints on the title screen, audio-unlock-on-first-keypress.
building-platformer-games — tuned controller recipe: variable jump height, coyote time, jump buffering, AABB collision resolution order, known-good tuning constants; uses the WIN state for goal/flag levels.
improving-game-quality — feel & correctness checklist (the only quality checklist — the orchestrator defers here rather than carrying its own): margins/readability, always-playable loop with reachable lose condition, ambient particles, shake on events, scene transitions incl. PAUSED/WIN, audio coverage, alt-tab resume, performance.
messaging-game-over — when/how to post state and score transitions via runtime.ts (thin — the module enforces the wire format).
adding-easter-egg — palette-swap toggles and secret input sequences (↑↑↓↓…).
playing-the-game — the dev-server lifecycle + smoke check, honest about what Claude can verify:
  1. Reclaim the port (one server at a time): kill whatever is listening on 5173 — lsof -ti:5173 | xargs kill, harmless when nothing is. Port-based, not handle-based, because background-task handles don't survive across Claude sessions: an orphan server from an ended session would otherwise hold the pinned port forever and deadlock every future launch.
  2. Launch in background (never foreground — Vite never exits and would hang the session): cd workspace/<game-name> && npm run dev as a background task; poll its output for the "Local: http://localhost:5173" readiness line. Port is pinned (5173, strictPort: true) so it fails loudly instead of silently drifting to 5174.
  3. Runtime smoke check: npm run smoke — the template's smoke.mjs drives headless Chromium (Playwright from root devDeps) against the dev URL, listening for console/page errors: asserts the page loads, a <canvas> exists in the live DOM after the module runs, and zero uncaught errors fired; exits nonzero on any failure. A real browser executes the JS — curl cannot make these assertions.
  4. Hand off: give the user the URL and the human playtest checklist (title renders, controls, pickups, hazard→shake→game over, restart, starfield, CRT, HUD margins). The user is the playtester — Claude cannot see the canvas and never claims otherwise.
  5. Teardown: stop the server when the user is done or before reset — same port-based kill as step 1, so it works even on a server this session didn't start.
resetting-the-workspace — destructive, so maximally guarded:
  1. Stop any running dev server — port-based (lsof -ti:5173 | xargs kill), so orphans from ended sessions are reclaimed too; never delete a folder out from under one.
  2. Enumerate game folders and confirm the exact list by name with the user; offer to keep (skip) or checkpoint-commit any game before wiping. A blanket "yes" without the named list shown is not confirmation.
  3. Allowlist delete: remove each confirmed folder by its exact path (rm -rf workspace/<name> per folder) — never a wildcard or exclusion glob that could match game-template. Guard: refuse unless CWD is the repo root and workspace/game-template exists both before and after.
  4. Verify: every folder on the confirmed delete-list is gone; only game-template and any explicitly-kept games remain; git status --porcelain workspace/game-template is clean. (The post-condition must honor the keep option — "only the template remains" is only correct when nothing was kept.)
Committed games remain recoverable from git history after a reset. Never touches the template, skills, or repo files — and now verifies that claim instead of assuming it.

05Validation gates — what "done" means

  1. Per edit: npm run check (tsc --noEmit) — fast inner loop, no bundling.
  2. Pre-handoff: npm run build (tsc --noEmit && vite build) passes.
  3. Pre-handoff: npm run smoke passes (headless-Chromium load: canvas in the live DOM, zero uncaught console/page errors).
  4. Actual playability: the user plays it. Claude reports "builds, boots clean, ready to play at <URL>" — never "playtested".

06Build order

  1. Scaffold: root package.json (devDeps: typescript, vite, playwright; no workspaces field), .gitignore, npm install, npx playwright install chromium, git init; ensure a git identity (set repo-local user.name/user.email if none is configured) — the whole recovery story depends on commits succeeding, so a failed baseline commit is a hard scaffold error, never skipped
  2. workspace/game-template/: package.json, tsconfig, vite.config.ts, smoke.mjs, arcade shell, engine modules, reference game
  3. Eleven skills + CLAUDE.md (conventions defined once; routing links to orchestrator) + README.md
  4. Tiny parent-frame harness page for verifying the runtime.ts postMessage path (driven by the same headless-Chromium mechanism as smoke.mjs — asserting received messages also requires executing JS)
  5. Initial git commit

07Verification

  • npm run build in a clone of the template passes (tsc --noEmit + vite build); no stray .js emitted next to sources; no node_modules/ created inside the game folder after npm run dev.
  • dist/ works when served from a subpath (proves base:'./' → independently deployable).
  • Human playtest of the reference game: title renders, controls work, audio unlocks on first keypress, pickups score, hazard triggers shake + game over, restart works, starfield + CRT visible, HUD respects margins.
  • Alt-tab 30s and return: game resumes without a freeze (catch-up spiral) or physics teleport.
  • runtime.ts: console messages locally and correct {source:'retrovibe', type, payload} messages received in the parent-frame harness.
  • End-to-end with Claude in the loop, three sample prompts: (a) a non-platformer creation, (b) a platformer creation, (c) an edit to an existing game ("make the ship faster"). Each must route to the right skills, build, and pass the smoke check.
  • Smoke check catches real breakage: npm run smoke passes on the healthy reference game and fails nonzero on a deliberately broken clone (e.g. a throw in main.ts) — proving the gate has teeth.
  • Orphan reclaim: with a dev server left running (simulating an ended session), the next launch reclaims port 5173 and succeeds instead of dying on strictPort.
  • Collision guard: attempting to create a game whose folder exists stops with a choice — never a nested-template corruption.
  • Reset round-trip: create → play → reset → only game-template (plus any explicitly-kept games) remains, template is git-clean, no orphaned dev server survives, and a checkpoint-committed game is recoverable from history. Also: reset keeping one game preserves exactly that game.
