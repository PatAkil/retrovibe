▚ Retrovibe — MVP Plan

Describe a retro game in Claude → skills turn it into a playable, high-quality game. Laptop creation loop: create → play → reset → repeat. Deploy comes post-MVP.

Settled decisions

Stack: TypeScript + Canvas 2D, no game framework. A small transparent engine in this repo encodes the quality rules as primitives; skills carry all game-making knowledge and all commands — no helper scripts. Every game is a self-contained folder cloned from a template:

cp -r workspace/game-template workspace/<game-name>

01Repo layout

retrovibe/
├── package.json              # devDeps installed ONCE at root: typescript, vite
├── node_modules/             # shared — game folders resolve bins by walking up
├── workspace/
│   ├── game-template/        # pristine + self-contained; never modified by game creation
│   │   ├── package.json      # scripts: dev, build
│   │   ├── tsconfig.json
│   │   ├── index.html        # arcade shell (cabinet chrome), mounts game/main.ts
│   │   ├── engine/           # copied with the template → each game standalone
│   │   └── game/             # minimal reference game
│   └── <game-name>/          # user's games — clones of game-template (wiped on reset)
├── .claude/skills/           # SKILL.md each — commands baked in
├── CLAUDE.md                 # repo map, workspace pattern, engine API, skill routing
└── README.md
Engine is copied into each game, not shared. Every game folder is standalone and independently deployable later (npm run build → dist/); later engine changes can never break older games; creation stays a pure cp -r. Canonical source lives in game-template/engine/.
One install at the root. Per-game npm run dev / npm run build work because npm resolves binaries from ancestor node_modules — no install step per game.
Git: git init + one initial commit as the clean-slate baseline. No remote, no further commits unless asked.

02Engine modules — quality rules as defaults, not conventions

ModuleContents
loop.tsFixed-timestep loop (accumulator), update(dt)/render split, pause on blur
input.tsUnified keyboard map (arrows+WASD → direction; Space/Z/X/Enter → actions), pressed/held/released edges, stuck-key handling on blur
scenes.tsEnforced state machine TITLE → PLAYING → GAME_OVER → restart — always playable, lose state always reachable
draw.tsPixel-scaled rendering, ASCII-art sprite maps → colored pixel sprites, retro text
palette.tsCurated retro palettes (PICO-8-style) + palette-swap support (feeds easter eggs)
particles.tsAmbient background presets (stars, rain, snow, embers, bubbles) + burst emitter for impacts
juice.tsScreen shake, screen flash, hit-stop / freeze frames
audio.tsWebAudio chiptune synth presets: jump, pickup, explosion, hit, UI blip — zero asset files
ui.tsHUD helpers (score/lives/text) with enforced SAFE_MARGIN from viewport edges
crt.tsRetro screen filter: scanlines + vignette + subtle flicker
runtime.tsHost contract: gameOver(score), scoreChanged(n), stateChanged(s) — postMessage to parent when embedded, console locally

Your four rules, mapped: UI margins → ui.ts always-playable loop + lose condition → scenes.ts background particles → particles.ts screen shake → juice.ts

03Template game

A minimal complete game (~150 lines) inside game-template/game/ proving every rule — and serving as the reference implementation all skills point to: title screen → play (move a ship, collect pickups, dodge a hazard; contact = lose) → game over → restart. Starfield, shake+flash+burst on hit, chiptune sfx, safe-margin HUD, CRT filter, runtime messaging.

04The skills

creating-a-game — the orchestrator, mirroring your flow:
  1. Setup workspace: folder-friendly name, cp -r workspace/game-template workspace/<game-name>; no dependency install needed.
  2. Develop: read the copied game files first; modify to fulfill the request. Use the companion skills: messaging-game-over, ensuring-arcade-visuals, handling-user-input, improving-game-quality, adding-easter-egg — and building-platformer-games if it's a platformer.
  3. Validation loop: after every edit, cd workspace/<game-name> && npm run build. Don't proceed or notify the user until compilation succeeds.
helping-the-user — behavior-elicitation questions, combining described behaviors into a coherent core loop, scoping guidance.
ensuring-arcade-visuals — palette discipline, pixel scale, CRT filter integration, readability rules.
handling-user-input — the unified keyboard contract, edge-vs-held semantics, control hints on the title screen.
building-platformer-games — tuned controller recipe: variable jump height, coyote time, jump buffering, AABB collision resolution order, known-good tuning constants.
improving-game-quality — the quality checklist: margins, always-playable loop with reachable lose condition, ambient particles, shake on events, transitions, audio coverage, performance.
messaging-game-over — when/how to post state and score transitions via runtime.ts (thin — the module enforces it).
adding-easter-egg — palette-swap toggles and secret input sequences (↑↑↓↓…).
playing-the-game — cd workspace/<game-name> && npm run dev, playtest in browser against the checklist.
resetting-the-workspace — reset baked in as a skill: list game folders, confirm with the user, delete everything in workspace/ except game-template. Never touches the template, skills, or repo files.

05Build order

  1. Scaffold: root package.json, npm install, git init
  2. workspace/game-template/: package.json, tsconfig, arcade shell, engine modules, reference game
  3. Ten skills + CLAUDE.md + README.md
  4. Initial git commit

06Verification

  • cd workspace/game-template && npm run build passes (tsc + vite build).
  • npm run dev → playtest: title renders, controls work, pickups score, hazard triggers shake + game over, restart works, starfield + CRT visible, HUD respects margins, console shows runtime messages.
  • Clone round-trip exactly as the skill prescribes: copy → build → play → reset → only game-template remains.
